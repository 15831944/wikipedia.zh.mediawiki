== 概述 ==
{{see also|指令流水线|乱序执行|推測執行|CPU快取|分级保护域|虚拟内存|分頁|記憶體保護}}
熔毀缺陷是基於設計不良的[[數位電路|數位電路]]設計中出現的[[竞争冒险|竞争条件]]，再結合許多現代[[微處理器|微處理器]]的[[乱序执行|非依序執行]]及[[推測執行|推測執行]]特性而產生。<ref>{{Cite web |url = https://www.hksilicon.com/articles/1515540 |title = Ben Thomposn：「幽靈」與「熔毀」漏洞映射出了科技行業的現狀 |author =  |date = 2018-01-10 |publisher = 香港矽谷 |language = zh-hk |accessdate = 2018-05-28 }}</ref>

在具備非依序執行及推測執行特性的處理器上，處理器會檢測[[指令|指令]]的相依性，對無相依性的指令會進行預先執行，或者當處理器執行某一指令出現異常而面臨停頓時，處理器會先執行與異常指令無相依性的指令以跳過停頓，這兩類指令執行方式的執行結果都會保存到CPU快取上一段時間以備使用（超過時間即拋棄結果），這些預測執行的動作包括分支预测，预读取，推测性内存访问，缓存缺失的重叠/乱序处理（MSHR）等等。<ref>{{cite web|title=开年惊吓！Intel承认CPU安全漏洞存在，厂商们一夜回到20年前|url=https://www.guokr.com/article/442607/|date=2018-01-08}}</ref>

某行程的某條指令需要其指定某個記憶體位址的資料進行運算，基於[[CPU快取|CPU快取]]的機制，該筆資料會先載入至CPU快取上，而這個過程對於指令而言是透明的，因此其它行程是無法直接從CPU快取上得知該筆資料的內容，而又因資料未載入至CPU暫存器上，需要該資料的指令仍無法得知資料內容，在這裏，基於[[分级保护域|分级保护域]]機制，CPU將資料從CPU快取載入至CPU的[[暫存器|暫存器]]前是需要經過資料權限檢查和記憶體位址合法性檢查，只有這些資料的存取權限被確認等於或低於該行程的權限、記憶體位址空間的存取權限符合該行程的權限時，方能被相應行程的指令存取，否則將資料丟棄並對該指令進行異常處理。但是，而從記憶體載入資料到CPU快取時，CPU是既沒有對這些資料進行權限檢查也沒有進行記憶體位址的合法性檢查的，而沒有被命中的CPU快取的資料也不會馬上被清空。如果CPU快取的資料內容無法被得知，那麼這種機制的處理邏輯並不會有問題，然而如果有方法可以從其它途徑「偷看」CPU快取的資料內容的話，那麼這種處理邏輯便存在缺陷。

要利用熔毀缺陷，在[[指令|指令]]处理期间，用異常的指令使得CPU進入對異常指令的異常處理當中，同時令CPU的推測執行去先執行一些非法指令，内存访问和權限检查或記憶體位址合法性檢查兩種操作在這裏發生竞争冒险，因为受影响的处理器中的[[指令流水线|指令流水线]]意味着在推测执行期间，来自未经授权地址的数据几乎总是被临时加载到CPU的缓存中，即使原始读取指令由于特权检查而最终失败并且从不产生可读结果，這個時候再利用[[旁路攻击|旁道攻击]]，利用存取CPU快取及存取記憶體的時間差，重建目標資料。這樣未经授权的进程就可从映射到当前进程的[[内存|内存空间]]的任何地址读取数据。<ref>{{cite web|title=处理器A级漏洞Meltdown（熔毁）和Spectre（幽灵）分析报告|url=http://www.freebuf.com/vuls/159269.html}}</ref>

由于许多操作系统将物理内存、内核进程和其他正在运行的[[用户空间|用户空间]]进程映射到每个进程的地址空间，并依靠特权检查来防止未经授权的访问，因此，熔毁缺陷有效地允许运行在用户空间的恶意进程在不需要权限提升的情况下，读取任何物理内核或其他进程的映射的内存，不管它是否应该能够这样做。若要防范这个缺陷所造成的危害，需要避免使用内存映射方式或使用内存分页隔离（均是基于软件的解决方案，后者是将用户空间的内存地址与内核空间的内存地址相隔离，但这将增加内核态与用户态之间的[[上下文交换|上下文交换]]开销），或避免潜在的竞争条件（修改CPU的[[微程序|微代码]]和/或执行路径）。

惡意程式利用熔毀缺陷進行攻擊活動時也無法被偵測感知，是該資安漏洞的一大危險之處。<ref name="spectreattack.com">{{cite web|url=https://spectreattack.com/ |title=Meltdown and Spectre |publisher=Spectreattack.com |date= |accessdate=2018-01-30}}</ref><ref>https://www.cybereason.com/blog/what-are-the-spectre-and-meltdown-vulnerabilities</ref>

更爲具體的，有缺陷的CPU型號數量非常多，Google的[[Project_Zero_(Google)|Project Zero]]報告指出，除了[[Itanium|安騰]]以及2013年前的[[Intel_Atom|Atom系列]]，英特爾自1995年發表基於[[P6微架構|P6微架構]]的[[Pentium_Pro|Pentium Pro]]開始所有具備非依序執行的CPU產品都有熔燬缺陷，<ref>{{cite web |url=https://www.bleepingcomputer.com/news/security/google-almost-all-cpus-since-1995-vulnerable-to-meltdown-and-spectre-flaws/ |title=Google: Almost All CPUs Since 1995 Vulnerable To "Meltdown" And "Spectre" Flaws |publisher=}}</ref>P6微架構是英特爾首款具備[[预测执行|预测执行]]的x86（[[IA-32|IA-32]]）微架構，首發產品便是Pentium Pro，而後續的英特爾x86微架構除Netburst(Pentuim 4)外都是以P6爲藍本。<ref>{{cite web |url=http://www.jaist.ac.jp/iscenter-new/mpc/altix/altixdata/opt/intel/vtune/doc/users_guide/mergedProjects/analyzer_ec/mergedProjects/reference_olh/reference_hh/p6_family_micro-architecture.htm |title=P6 family microarchitecture |website=www.jaist.ac.jp}}</ref>

而ARM表示它們的主要處理器IP核不受影響，具體受影響的IP核型號已經公佈。[[ARM_Cortex-A75|ARM Cortex-A75]]除了有熔毀缺陷以外還受幽靈漏洞的影響，而其它型號則沒有熔毀缺陷。帶預測執行技術的[[ARM_Cortex-R7|Cortex-R7]]、[[ARM_Cortex-R8|Cortex-R8]]、[[Cortex-A8|Cortex-A8]]、[[Cortex-A9|Cortex-A9]]、[[Cortex-A15|Cortex-A15]]、[[Cortex-A17|Cortex-A17]]、[[Cortex-A57|Cortex-A57]]、[[Cortex-A72|Cortex-A72]]及[[Cortex-A73|Cortex-A73]] IP核只受幽靈的影響。<ref name=":2">{{cite web |url=https://developer.arm.com/support/security-update |title=Arm Processor Security Update |publisher=[[ARM_Ltd.|ARM Ltd.]] |website=ARM Developer |date=January 3, 2018 |access-date=January 5, 2018}}</ref>ARM是在英特爾之後第二個被發現有熔毀缺陷的產品的，但與英特爾的不同，ARM唯一有熔燬缺陷的IP核產品Cortex-A75尚未有實作商用。<ref>{{cite web |url=http://fortune.com/2018/01/04/meltdown-spectre-intel-amd-arm-security-bug-apple-microsoft-google-apple-amazon/ |title=Understanding Those Alarming Computer Chip Security Holes: 'Meltdown' and 'Spectre' |publisher=}}</ref>另外，不使用預測執行技術的ARM架構處理器，也是當前使用數量最多的ARM產品，不受任何影響，也沒有熔燬缺陷，這部分包括了大量使用在Android行動裝置上的[[Cortex-A53|Cortex-A53]] IP核，以及即將有商用實作案的[[Cortex-A55|Cortex-A55]]。僅使用Cortex-A53、Cortex-A55的處理器產品包括[[高通|高通]]的[[高通驍龍|驍龍]]630、626/625、400系列等、[[聯發科技|聯發科]]的MT6795等、[[树莓派|樹莓派單板電腦]]使用的[[博通_(公司)|博通]]SoC等等。<ref>{{cite web |url=https://www.androidauthority.com/meltdown-spectre-kpti-827527/ |title='Spectre' and 'Meltdown': New CPU vulnerabilities affect most smartphones and computers |date=January 4, 2018 |publisher=}}</ref><ref>{{cite web|url=https://www.raspberrypi.org/blog/why-raspberry-pi-isnt-vulnerable-to-spectre-or-meltdown/ |title=Why Raspberry Pi isn't vulnerable to Spectre or Meltdown |publisher=Raspberry Pi |date=2018-01-05 |accessdate=2018-01-30}}</ref>

IBM的POWER架構是第三個被發現有熔燬缺陷、受幽靈的處理器微架構，[[紅帽公司|紅帽公司]]在2018年1月3日公佈了受這些缺陷影響的處理器型號，涵蓋[[IBM_System_Z|Z架構]]、POWER架構（包括最新的[[POWER8|POWER8]]及[[POWER9|POWER9]]）的產品，並發表了適用於這些指令集架構編譯版本的[[RHEL|RHEL]]之修復程式；IBM也立即公佈了受影響型號清單並發佈了相關的韌體修復程式和AIX、z/OS修復程式。<ref>{{cite web|last=Tung |first=Liam |url=http://www.zdnet.com/article/meltdown-spectre-ibm-preps-firmware-and-os-fixes-for-vulnerable-power-cpus/ |title=Meltdown-Spectre: IBM preps firmware and OS fixes for vulnerable Power CPUs |publisher=ZDNet |date=2018-01-10 |accessdate=2018-01-30}}</ref>

[[甲骨文公司|甲骨文公司]]表示[[SPARC|SPARC]] V9的系統（T5、M5、M6、S7、M7、M8、M10、M12處理器）沒有熔燬缺陷，但並未說明更老的、不再進行技術支援的處理器產品是否受這兩大硬體缺陷的衝擊。<ref>{{Cite news|url=https://talesfromthedatacenter.com/2018/01/solarissparc-is-meltdown-cve-2017-5754-free/|title=Solaris+SPARC is Meltdown (CVE-2017-5754) free - Tales from the Datacenter|date=January 22, 2018|work=Tales from the Datacenter|access-date=January 23, 2018|language=en-US}}</ref>

==漏洞修复==
2018年初，微软发布了新的[[Windows|Windows]]系统的安全补丁以修复漏洞。英特尔公司多次表示修复漏洞并不会过多影响芯片性能，但微软的测试表明在安装了漏洞补丁的[[Windows_7|Windows 7]]与[[Windows_8|Windows 8]]环境下，2015年及以前生产的英特尔芯片会有明显可感的性能下降，有消息说下降幅度甚至可以达到30%，较旧的[[Haswell|Haswell]]架构与之前的芯片性能下降尤为明显。但在使用[[Skylake|Skylake]]、[[Kaby_Lake|Kaby Lake]]等较新架构的英特尔芯片的[[Windows_10|Windows 10]]系统上，性能下降则并不明显。<ref>[https://www.moneydj.com/KMDJ/News/NewsViewer.aspx?a=fb5e7bc2-42be-4465-9619-de0b8964b582  安全漏洞作祟、微軟：更新會拖慢電腦]</ref>
